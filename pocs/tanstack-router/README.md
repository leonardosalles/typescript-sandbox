# TanStack Router — POC

## Stack

- **Vite** + **React 18** + **TypeScript**
- **TanStack Router v1** (file-based routing)
- **Zod** (search param validation)
- No UI library — custom design system

---

## Como rodar

```bash
pnpm install
pnpm run dev
# Acesse http://localhost:5173
# Login: admin@poc.dev / admin123
```

## Features Exploradas (Deep)

### 1. File-Based Routing

O plugin `@tanstack/router-vite-plugin` observa `src/routes/**/*.tsx` e gera `routeTree.gen.ts` automaticamente.

Convenções:

- `__root.tsx` → rota raiz (sempre renderizada)
- `index.tsx` → `/dashboard/` (trailing slash = index)
- `$userId.tsx` → parâmetro dinâmico `:userId`
- `_reports.lazy.tsx` → underscore = não registrado como rota (lazy chunk)
- `dashboard/` → pasta = segmento de path

**Por que isso é melhor que registrar rotas manualmente?**
Porque o tipo do routeTree é gerado automaticamente. `<Link to="/dashboard/users/$userId" params={{ userId: '1' }}>` é 100% type-safe sem nenhuma anotação manual.

### 2. RouterContext — Injeção de Dependência em Loaders

```ts
// types/router.ts
export interface RouterContext {
  auth: ReturnType<typeof getAuth>;
}

// main.tsx
const router = createRouter({
  routeTree,
  context: { auth: getAuth() },
});

// posts/index.tsx
loader: async ({ context }) => {
  console.log(context.auth.user?.email); // typed, injected
};
```

**Por que isso importa?**
Loaders são funções puras — não são componentes React. Eles não podem usar hooks. `RouterContext` é o único mecanismo oficial para passar dependências externas (auth, queryClient, feature flags) para dentro dos loaders sem singletons globais. Isso torna os loaders **testáveis** — você pode injetar um mock de `auth` nos testes.

### 3. Search Params com Zod — A Feature Mais Poderosa

```ts
// Declara uma vez, usa em todo lugar
const usersSearchSchema = z.object({
  page: z.number().int().min(1).default(1),
  search: z.string().default(""),
  role: z.enum(["", "admin", "editor", "viewer"]).default(""),
  sortBy: z.enum(["name", "createdAt", "postsCount"]).default("name"),
});

export const Route = createFileRoute("/dashboard/users/")({
  validateSearch: usersSearchSchema,
  loader: async ({ search }) => {
    // search.page é number, não string. Zod coerce automaticamente.
    const result = await fetchUsers(search);
    return { result };
  },
});
```

**Por que URL > useState para filtros:**

1. **Compartilhável**: `/users?role=admin&sortBy=postsCount` — manda o link pro colega
2. **Bookmarkável**: bookmark da sua view favorita funciona
3. **Browser history**: back/forward funciona corretamente com filtros
4. **Zero sync bugs**: estado da UI e URL nunca divergem (impossível)
5. **SSR-friendly**: servidor lê filtros da URL sem hydration mismatch

**Como atualizar search params:**

```ts
const navigate = useNavigate({ from: Route.fullPath })

// Reset page quando filtro muda
navigate({ search: (prev) => ({ ...prev, role: 'admin', page: 1 }) })

// Via Link — hover preloads o loader
<Link to="/dashboard/users" search={(prev) => ({ ...prev, page: 2 })}>
  Next
</Link>
```

### 4. Route Loaders — Data First, Component Second

```ts
export const Route = createFileRoute("/dashboard/")({
  loader: async () => {
    const stats = await fetchDashboardStats();
    return { stats }; // tipo inferido automaticamente
  },
  staleTime: 10_000, // cache por 10s
  pendingComponent: Skeleton, // mostrado durante o load
  component: DashboardPage,
});

function DashboardPage() {
  const { stats } = Route.useLoaderData(); // garantido não-null
  // ...
}
```

**Padrão de fetch paralelo:**

```ts
loader: async ({ params }) => {
  // NÃO faça isso: 400ms + 500ms = 900ms
  const user = await fetchUser(params.userId);
  const posts = await fetchPosts();

  // FAÇA isso: max(400ms, 500ms) = 500ms
  const [user, posts] = await Promise.all([
    fetchUser(params.userId),
    fetchPosts(),
  ]);
  return { user, posts };
};
```

**`staleTime`**: se você navegar pra outra rota e voltar em menos de 10s, o loader não re-executa. Cache grátis, sem React Query.

### 5. `beforeLoad` — Guards e Autorização

```ts
// Autenticação
beforeLoad: () => {
  if (!isAuthenticated()) {
    throw redirect({ to: "/login" });
  }
};

// Autorização por role
beforeLoad: () => {
  if (!isAuthenticated()) throw redirect({ to: "/login" });
  if (!hasRole("admin")) {
    throw redirect({
      to: "/dashboard",
      search: { error: "insufficient_permissions" },
    });
  }
};

// Login — redireciona SE já autenticado
beforeLoad: () => {
  if (isAuthenticated()) {
    throw redirect({ to: "/dashboard" });
  }
};
```

**`throw redirect()` vs `return redirect()`**: TanStack Router usa `throw` para interromper o pipeline de loading imediatamente. Análogo ao `throw promise` do React Suspense — é uma saída de controle de fluxo, não um retorno normal.

### 6. `lazyRouteComponent` — Code Splitting Natural

```ts
// reports.tsx (sempre no bundle principal — pequeno)
export const Route = createFileRoute("/dashboard/reports")({
  loader: async () => {
    // Loader e download do chunk rodam em PARALELO
    const stats = await fetchDashboardStats();
    return { stats };
  },
  component: lazyRouteComponent(
    () => import("./_reports.lazy"),
    "ReportsPage", // named export
  ),
});

// _reports.lazy.tsx (chunk separado — pode ter dependências pesadas)
export function ReportsPage() {
  /* ... */
}
```

**Como verificar o code splitting:**

1. Abra Network tab
2. Navegue para fora de Reports
3. Hover no link "Reports (Lazy)" — chunk baixa antes do click (`preload="intent"`)
4. Click — navegação instantânea, sem spinner

**Por que splits em rotas são superiores a `React.lazy`:**

- Ponto de split natural — usuário sinalizou intenção ao navegar
- Loader + download do chunk em paralelo
- `pendingComponent` lida com o estado de loading de forma elegante
- TypeScript funciona normalmente — o componente lazy é tipado

### 7. `errorComponent` — Error Boundaries Granulares

```ts
export const Route = createFileRoute('/dashboard/users/')({
  loader: async ({ search }) => {
    // api.ts tem 5% de chance de falhar — testa errorComponent
    const result = await fetchUsers(search)
    return { result }
  },
  errorComponent: ({ error, reset }) => (
    <div>
      <ErrorDisplay error={error} retry={reset} />
    </div>
  ),
})
```

**Por que isso é arquiteturalmente importante:**

- Apenas a rota com erro quebra — sidebar, nav e outras rotas continuam funcionando
- `reset()` re-executa o loader — retry automático
- Cada rota pode ter seu próprio tratamento de erro contextual
- Contrasta com React ErrorBoundary manual que você tem que lembrar de adicionar

---

### 8. `pendingComponent` — Skeleton Colocado com o Loader

```ts
export const Route = createFileRoute('/dashboard/users/')({
  loader: fetchData,
  pendingComponent: () => <SkeletonTable rows={5} cols={5} />,
  component: UsersPage,  // só renderiza quando loader termina
})
```

**Benefício arquitetural**: o componente `UsersPage` não precisa lidar com estados de loading. `useLoaderData()` sempre retorna dados prontos. Isso é separação de responsabilidades real — o loader cuida de dados, o componente cuida de apresentação.

## Experimentos para ir mais Fundo

### Experimento 1: Route Invalidation

Após uma ação de mutação (ex: salvar settings), force o re-fetch dos loaders:

```ts
const router = useRouter();
await saveSettings(data);
router.invalidate(); // re-roda todos os loaders ativos
```

### Experimento 2: Preloading Programático

Pré-carregue dados de uma rota antes do usuário navegar:

```ts
const router = useRouter();
// Em um onMouseEnter:
router.preloadRoute({
  to: "/dashboard/users/$userId",
  params: { userId: "1" },
});
```

### Experimento 3: Search Params Cross-Route

Search params são por-rota. Para compartilhar estado entre rotas, use o hash ou params no path.
Experimento: adicione um filtro global (ex: `?theme=dark`) no root route e observe como se propaga.

### Experimento 4: Deferred Data (Streaming)

```ts
loader: async () => {
  const criticalData = await fetchCritical();
  const slowData = fetchSlow(); // NÃO await — promise, não valor

  return {
    criticalData,
    slowData: defer(slowData), // renderiza agora, aguarda depois
  };
};
```

O componente usa `<Await resolve={loaderData.slowData}>` para mostrar loading progressivo.

### Experimento 5: Search Params com Objetos Complexos

Zod + TanStack Router suportam arrays e objetos em search params:

```ts
const schema = z.object({
  filters: z.array(z.string()).default([]),
  dateRange: z.object({ from: z.string(), to: z.string() }).optional(),
});
```

Experimento: como eles são serializados na URL? O que acontece com `encodeURIComponent`?
